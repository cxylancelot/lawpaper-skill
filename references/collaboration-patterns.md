# 协作场景模式库

> 6 种最常见的协作场景，每种给出标准的协作流程。当你不确定该怎么推进时，加载此文对照当前场景。

---

## 场景 1：用户有开题报告但写不出论文正文

**典型状态**：用户上传了开题报告（或口述了开题内容），说"我开题通过了，但正文写不出来"。

**协作模式**：拆解→追问→重构→逐章展开

**流程**：

```
1. 🔍 诊断
   - 通读开题报告，提取：研究对象、核心问题、论点方向、拟用方法、文献清单、大纲
   - 找出开题报告中最弱的部分（通常是"问题意识不够尖锐""论证路径不清楚""方法不匹配"）
   - 向用户反馈诊断："你的开题报告在X方面是清楚的，但在Y方面还需要在写作中进一步明确。
     我建议我们先把Y厘清，然后从[某个章节]开始写。"

2. ❓ 追问（聚焦开题的缺口）
   - 逐个追问开题中没写清楚的部分
   - 每次只问一个问题，等用户回答
   - 示例：
     "你的开题说'本文采用案例分析的方法'，但你选了哪些案例、选它们的标准是什么？"
     "你的核心观点是[开题中的某句]，但如果有人说'这完全是另一种原因造成的'，你怎么反驳？"

3. 💬 讨论
   - 在追问过程中如果用户犹豫，切换到讨论模式
   - 按方法论分析每个选项的利弊

4. ✍️ 下笔：从选题陈述开始
   - 先重写"选题陈述"（一段话浓缩论文核心），确认方向对齐
   - 然后按 Step 5 的结构逐章下笔：
     → 大纲 → 引言 → 第一章 → 第二章 → 第三章 → 结论 → 摘要
   - 每章一个循环（诊断已有材料→追问缺口→下笔→确认）

5. 🔄 回顾
   - 写完一章后，检查是否与选题陈述一致
   - 如果写作中论点发生偏移（正常的），回到选题陈述更新它
```

**注意事项**：
- 开题报告≠正文大纲。很多开题的"大纲"实际是不可执行的（全是标题没有论证功能）。不要按开题大纲硬写。
- 开题中承诺的内容在写作中可能发现不可行——诚实地和用户讨论调整方案，不要硬着头皮按开题写。

## 场景 2：用户只有大致方向，无开题无材料

**典型状态**：用户说"我想写关于XX的论文，但不知道怎么开始"。

**协作模式**：决策树→讨论→选题陈述

**流程**：

```
1. 🔍 诊断
   - 加载 decision-tree.md
   - 从用户的表述中提取：研究起点是什么？

2. ❓ 追问
   - 跑 Step 0 的提问引导（论文类型→研究对象→核心困惑→已有材料）
   - 如果用户答不上"核心困惑"：
     "没关系，我们换个角度。你读过的跟这个方向相关的材料里——
      法条、判决、论文——有没有什么地方让你觉得'这不对''这不合理''这怎么解释'？"
   - 由此定位问题意识的潜在方向

3. 💬 讨论
   - 如果用户的方向太宽（"我想写民法典"），讨论收窄策略
   - 如果用户有几个可能方向，按三维创新判断（规范增量/实践增量/理论增量）逐个分析

4. ✍️ 下笔：选题陈述
   - 当问题清晰到可以一句话说清时，写出"选题陈述草稿"
   - 用户在选题阶段看到一段清晰陈述后，通常能判断"这就是我想写的"或"不对，我想的是另一个方向"
   - 如果是后者，回到追问，调整方向

5. 过渡到下一步
   - 选题确认后，建议："下一步我们做文献检索。你有任何已知的相关文献吗？
     没有的话，告诉我几个关键词，我给你检索策略。"
```

**注意事项**：
- 在用户没有材料没有开题的阶段，不要急着下笔写正文——先把方向定清楚。
- 选题陈述写出来后，如果用户说"好像没什么新意"，冷静地分析为什么（可能确实没新意，也可能是用户低估了自己的贡献）。

## 场景 3：用户写了初稿但不满意

**典型状态**：用户上传了完整或部分草稿，说"这稿子不行，但我说不出哪里不行"。

**协作模式**：诊断→定位问题→讨论→重写

**流程**：

```
1. 🔍 诊断
   - 通读草稿，按照对应阶段的方法论逐项诊断
   - 先找出最致命的问题（通常是论证层面：核心论点不清晰、论据不足、逻辑跳跃）
   - 再找出次要问题（引注格式、表达口语化等）
   - 向用户反馈专业诊断：
     "我读了你的草稿，发现了几个层面的问题——
      最需要先解决的是[致命问题]，这会影响整篇论文。
      其次是[次要问题]，这些我们可以在修完核心问题后再处理。
      我们先讨论[致命问题]，你看可以吗？"

2. ❓ 追问
   - 围绕致命问题追问：
     "这篇论文的核心论点是什么？你心里想的，不一定是你写出来的——你能用一句话说给我听吗？"
     "如果核心论点是X，你论文的第三章好像论证了一个相反的Y。是你真的这样认为，还是写的时候走偏了？"

3. 💬 讨论
   - 讨论修改方案：是局部修改还是结构性重写？
   - 如果问题太深（如核心论点根本站不住），诚实告知而不是粉饰

4. ✍️ 下笔：改写
   - 按优先级逐段改写
   - 每次下笔附带教练批注，说明改了什么、为什么这样改、和原来有什么区别

5. 🔄 回顾
   - 改写完成后，检查是否解决了最初定位的所有问题
```

**注意事项**：
- 诊断越具体越好。"这段论证有问题"不如"这段从A跳到C，缺了'因为A所以B'的推理步骤"。
- 如果问题太多，不要一次全说——挑最重要的2-3个，先解决再往下。

## 场景 4：用户卡在某一个段落，写不下去

**典型状态**：用户说"引言我写到第二段就写不动了""论证部分我不知道该怎么展开"。

**协作模式**：追问材料→确认功能→下笔

**流程**：

```
1. 🔍 诊断
   - 读用户已经写出来的部分
   - 判断卡住的原因：信息不够？不知道这段的论证功能是什么？完美主义？

2. ❓ 追问
   - "这一段（或这个章节）在整个论文里是要完成什么任务？
     接收上一章的什么结论？为下一章打开什么问题？"
   - "写这一段需要什么材料？法条？判例？学说？你已经有什么？还缺什么？"

3. ✍️ 下笔
   - 确认了这段的论证功能和可用材料后，直接写出草稿
   - 标注所有不确定的信息
   - 附带教练批注："这段写的逻辑是……如果你觉得方向不对，可以调整为……"

4. 如果用户还不满意
   - 进一步追问："是方向不对，还是表达不对，还是你觉得信息不够？"
   - 针对性调整后重写
```

**注意事项**：
- 不要等用户把所有材料都准备好——有什么先用什么，缺的用占位符。
- 有时候用户不是不知道怎么写，而是完美主义作祟——告诉他"先写出来再改"。

## 场景 5：用户说"帮我改这一段"

**典型状态**：用户提供一段已有文字，要求帮忙修改润色。

**协作模式**：诊断→讨论→改写→解释

**流程**：

```
1. 🔍 诊断
   - 找出这段的问题，按性质分类：
     a. 论证薄弱（缺论据、逻辑跳跃、引而不证）
     b. 表达问题（口语化、重复、表述模糊）
     c. 结构问题（这段的论证功能和它在论文中的位置不匹配）
     d. 引注问题（缺出处、格式错误）
   - 向用户反馈诊断

2. 💬 讨论
   - 如果问题是 a 或 c（涉及论证），需要和用户讨论
   - 如果问题是 b 或 d（纯语言/格式），可以直接改

3. ✍️ 下笔：改写
   - 输出改写后的段落
   - 附带教练批注，逐条说明改了什么：
     "原稿的问题：……
      我做了什么改动：……
      为什么这样改：……
      如果你觉得改动太大，这里有一个只改最小幅度的版本：……"

4. 用户不满意 → 回到步骤 2，讨论调整方向
```

**注意事项**：
- 改写不是"我觉得这样更好"——每一次改动都要有方法论依据。
- 如果用户要求"学术化润色"，加载 `references/academic-expressions.md` 对照替换。

## 场景 6：用户说"按这个大纲帮我写一章"

**典型状态**：用户给了大纲和各章预设内容，说"帮我写第三章"。

**协作模式**：确认结构→逐节协作

**流程**：

```
1. 🔍 诊断
   - 不盲从大纲。先检查用户给的大纲是否合理：
     - 章节之间的关系是递进还是并列？
     - 这一章的论证功能是否明确？
     - 这一章在大纲中的篇幅和它的论证重要性匹配吗？
   - 如果有结构问题，先讨论再调整

2. ❓ 追问
   - "这一章在你的论证链条里，解决什么具体问题？"
   - "你手头有什么材料是专门为这一章准备的？"

3. ✍️ 下笔
   - 将这一章拆成若干节，逐节协作（每节独立：追问→下笔→确认）
   - 每节注意段落的论证功能单一原则
   - 标注所有不确定信息

4. 全章写完后
   - 回顾这一章：和上一章连接顺畅吗？为该章开始的论证张力收尾了吗？
```

**注意事项**：
- 用户的大纲可能是"开题时随便写的"——动手写之前一定要检查。
- 如果这一章篇幅失控（"提纲说写3000字但写出来可能要8000"），和用户讨论是调整篇幅还是拆分章节。

## 跨场景通用原则

### 1. 不跳跃
不跳过追问直接下笔。不跳过诊断直接给建议。

### 2. 每次只做一件事
不一口气诊断+追问+下笔全输出。一次一个行为，等用户回应。

### 3. 下笔必附批注
每个草稿输出必须告诉你"为什么这样写"和"风险在哪"。这不仅帮你理解决策，也让你在提交论文时能向导师/审稿人解释你的选择。

### 4. 尊重用户的最终决定
我分析利弊，你做出判断。当用户做了我方法论框架下"不推荐"的选择，我会说明风险和理由，但尊重你的判断——论文是你的。

### 5. 透明标注不确定性
下笔输出中可以区分三种信息：
- 确定的内容（基于你提供的信息+教练方法论的推理）
- 不确定但可生成的内容（方法论上合理，但需你确认——标注 `[待确认]`）
- 完全不能生成的内容（信息完全缺失——标注 `[待用户提供]`）
